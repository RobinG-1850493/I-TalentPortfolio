### OpenRPIGames

De bedoeling van deze opdracht is twee raspberry pi’s blad steen schaar tegen elkaar laten spelen. Dit gaat in verschillende fases uitgewerkt worden. Eerst op basis van kleur, dan op basis van vormen. Ik deel deze opdracht met Niels Debrier, hij gaat het vorm gedeelte voor zijn rekening nemen terwijl ik voor het kleur gedeelte zorg.

De code kan teruggevonden worden op de volgende github pagina: https://github.com/PXLRoboticsLab/OpenRPiVisionGames/tree/RG_co

De doelstelling is dus twee raspberry pi’s tegen elkaar blad steen schaar te laten spelen. We gaan dit doen op basis van kleuren, het idee hier achter is dat we elke onderdeel een kleur geven. Dus blad is bijvoorbeeld rood, steen is blauw en schaar is groen. Mijn doel is eerst en vooral uitzoeken welke bestaande methodes hiervoor gebruikt kunnen worden en deze implementeren. Nadat we een werkend proof of concept hebben wil ik een eigen methode ontwikkelen en deze vergelijken met de andere methodes.

Ik heb een aantal methodes bekeken en uit geprobeerd. De eerste was een python library die toeliet kleuren te herkennen binnen foto’s. Het probleem met deze methode was dat de library enkel werkte op opgeslagen foto’s. Dit betekende dat elke frame apart moest opgeslagen worden en dan pas geanalyseerd kon worden waardoor de performance niet acceptabel was. Deze methode werkte effectief wel maar de uiteindelijk gebruikte oplossing was een stuk sneller en simpeler.

De tweede methode was het gebruik van k-means clustering. Deze methode bekeek het beeld en gaf een histogram terug van percentages. Deze percentages correspondeerde met de hoeveelheid van het beeld welke kleur had. Jammer genoeg was deze methode ook niet geschikt omdat er geen mogelijkheid was om de kleuren te definiëren in code. We konden dus niet zeggen dat het percentage bijvoorbeeld rood was.  

Mijn eigen idee is niet het exact bepalen van welke kleur het is maar drie kleur ranges bepalen. En dan kijken of de kleur binnen deze range valt en op basis hiervan beslissen welke actie moet ondernomen worden. Hieronder is de code die dit verwezenlijkt:
 
Figure 1: De get_colour methode die bepaald welke actie moet ondernomen worden
We gaan dus voor elke kleur (rood, groen en blauw) een range bepalen, de blue range bevat dus alle blauwe kleuren. Met behulp van deze ranges gaan we masks maken die enkel de pixels bevatten die binnen deze range vallen. Deze masks zijn dus rood, groen of blauw. Deze masks zijn eigenlijk gewoon een grote array van pixel values, deze values zijn representeren de kleur van de pixel. Als er minstens 1 pixel binnen deze mask niet 0 is betekent dit dat die kleur effectief voor komt binnen het beeld. 

We weten dat de raspberries het volledige beeld 1 kleur geven dus kunnen we er van uit gaan dat in twee van de drie masks alle pixel values nul zijn. Dus uiteindelijk weten we in welke kleuren range het beeld valt. Nu moeten we enkel nog aan elke kleuren range een value (Rock, Paper of Scissors) toekennen. Deze manier is een stuk sneller en simpelere dan de eerder gebruikte methode.

Nadat de functionaliteit klaar was heb ik deze in het interface geïmplementeerd dat door Niels Debrier gebruikt werkt en de nodige aanpassingen gemaakt zodat het op rasberry pi kan gebruikt worden.

Op de raspberries zelf moet OpenCV geïnstalleerd worden, dit vormde een probleem omdat OpenCV moet gecompileerd worden, dit is een zeer intensieve operatie die de raspberry niet aankon. De oplossing hiervoor was ROS te installeren, ROS bevat een op voorhand gecompileerde versie van OpenCV.
Reflectie
Tijdens deze opdracht heb ik vrij veel problemen ondervonden omdat alle bestaande methodes eigenlijk ongeschikt waren voor deze use case en dus heb ik mijn eigen methode moeten bedenken. Deze gebruikte methode was uiteindelijk een zeer simpele maar ook effectieve oplossing die exact deed wat er gevraagd werd.  Een tweede probleem was de installatie van OpenCV op de raspberry pi. Een oplossing voor dit probleem werd gevonden door overleg met verschillende studenten. De uiteindelijke oplossing was ROS installeren om gebruik te maken van bijgevoegde versie van OpenCV.

Dankzij de nood aan een zelf bedachte methode was een zeker een tof project, zeker om het logisch redeneren en uitwerken te oefenen. De samenwerking met Niels was ook een plus punt. We hadden beide besproken wat de bedoeling was en wie wat ging doen. 

Een zwak punt tijdens het project was dat ik te lang stil stond bij bestaande methodes en toch een manier probeerde te zoeken om deze bruikbaar te maken. Het logisch denken en redeneren bij het bedenken en uitwerken van deze methode was dan weer een sterk punt.

### HoloLens Experiments

Het idee achter deze opdracht is kennismaken met de HoloLens en Unity. Dit ter voorbereiding van een project in opdracht van Scapta. De doelstelling is de basis principes van de HoloLens te begrijpen en hoe er mee om te gaan wanneer we gaan ontwikkelen.

Alle Code kan en werk informatie kan gevonden worden op de volgende github pagina: https://github.com/RobinGoos/HoloLensExperiments

Om de doelstelling te bereiken heb ik er voor gekozen de officiële tutorial van Microsoft te door lopen (https://docs.microsoft.com/en-us/windows/mixed-reality/holograms-101). Deze gaat over alle belangrijke aspecten van de HoloLens en hoe je er best mee omspringt. 

Voor dat ik kon starten met de HoloLens moest ik enkele software installaties doen. Visual Studio 2017, Unity 2017.4, Vuforia, HoloLens Emulator en de HoloToolkit. Vervolgens moest de HoloLens met de development environment geconnecteerd worden. In Unity gebeurd dit door een app te installeren op de HoloLens die dan voor connectie kan zorgen. In Visual Studio gebeurd dit door de HoloLens een pairing code te laten generen en deze in te geven wanneer men voor het eerst connecteerd.

Deel 1 van de tutorial leert ons hoe we Unity en Visual Studio correct configureren voor het gebruik van de HoloLens. Het doel van dit hoofdstuk is een hologram plaatsen en hiermee interactief mee kunnen omgaan. Het omgaan gebeurt met behulp van Gaze en Gestures. Gaze is eigenlijk waar de gebruiker naar toe kijkt, het is belangrijk dat wanneer de user naar onze hologram kijkt er ook iets gebeurd. We voegen highlighting toe aan de hologram wanneer de gebruiker er naar kijkt. Men kan dan met het airtap gesture de hologram beïnvloeden. Je kan bijvoorbeeld met de hologram gooien. 

 
Figure 1: Code voorbeeld -- Gaze manager

In deel 2 van de tutorial gaan we specifieker in op Gaze en op Gestures. We maken gebruik van een prefab hologram die uit verschillende onderdelen bestaat. Op basis van waar de gebruiker aan het kijken is verandert de kleur van het onderdeel. Als men dan het airtap gesture gebruikt verandert de kleur van het onderdeel permanent. We voegen ook een Voice command toe. Wanneer we “Reset Hologram” zeggen moet de hologram en al zijn onderdelen terug gaan naar de orginele kleur.

In het derde en laatste deel leren we hoe we de ruimtelijke capaciteiten van de HoloLens kunnen gebruiken. We gaan een hologram plaatsen, deze hologram speelt geluid af. Het geluid moet afkomstig zijn van de plaats van de hologram. Dus als we met ons linker oor naar de hologram toestaan moet de linkerkant luider zijn. Verder maken we gebruik van spatial mapping. De HoloLens scant de ruimte constant en geeft ten allen tijden informatie door over deze ruimte. We gaan deze informatie gebruiken om een wireframe te tekenen over de ruimte zodat we in de HoloLens de grenzen kunnen zien. 

Als opdracht om te kijken dat ik het effectief snapte had ik als doel een scan maken van het Dijkstra lokaal @ISpace en deze inladen in de HoloLens. Dit heb ik gedaan door te connecteren met het Windows device portal. Hier kan je een ruimte inscannen en dan als gameobject downloaden. Na het inscannen importeer ik de file in Unity en heb ik een gedetailleerde wireframe van het lokaal. Aan deze wireframe heb ik dan highlighting op basis van Gaze locatie toegevoegd en men kan ook hologrammen plaats met gestures. 

Om toch kort te illustreren dat deze technologie echt wel impressionant kan zijn, kijken we even naar de scan en de hoeveelheid detail in deze scan. De SanBot in de hoek van de ruimte is duidelijk herkenbaar in de wireframes net zoals alle schermen en alle andere objecten.

#### Reflectie

Dit project heeft me geleerd dat voorbereiding erg belangrijk kan zijn. Bij de start voelde ik toch dat ik al lange tijd niet meer met .NET had gewerkt en verfrissing echt nodig was. Verder bevestigd dit project dat mijn motivatie uit interesse voort komt en dus ben ik ook erg gemotiveerd voor het komende project bij Scapta. 

Qua sterke punten merk ik dat ik heel snel ben in het oplossen van eventuele bugs of problemen. Dit komt volgens mij door mijn denk proces wanneer er iets mis loopt. Ik lijst eerst alle mogelijkheden op en dan ga ik op basis van welke het meest waarschijnlijk is kijken waar de fout ligt. Ook mijn motivatie vond ik zeker een sterk punt. Een zwakker punt is het bijhouden van een verslag en planning van wat ik ga doen en wat ik gedaan heb. Dit zorgde soms voor verwarring bij de start van de dag.

### HoloDynamics365

De bedoeling van dit project is een applicatie ontwikkelen die een welcomings experience aan de klant geeft. Wanneer de klant binnenkomt ligt er een HoloLens klaar, wanneer men deze opzet krijgt de klant een overzicht van alle producten (als hologrammen rondom hem/haar). Als men op een product tapt opent een nieuw menu dat alle klanten van Scapta voor dit product laat zien. Wanneer men dan vervolgens op een klant tapt begint er een fimpje over deze klant te spelen (of een case-story pdf). Alle data moet uit het Dynamics 365 systeem van Scapta komen om zo dynamisch alle menu’s en videos te genereren.

Alle code van het project kan men terug vinden op de volgende twee github repositories.
-	HoloDynamicsAPI (Privaat)
-	HoloDynamics365 :: https://github.com/RobinGoos/HoloDynamics365

Voor ik aan het project begon heb ik met de klant (Scapta) samengezeten om de Scope van het project te bepalen. Dit was belangrijk om de verwachting uit te lijnen en een planning op te stellen. Hierdoor had ik een goede basis om van start te kunnen gaan.

 
Figuur 1: Werkplaats en de hololens
Voor ik aan het project begon moest ik er voor zorgen dat Unity en Visual Studio juist geconfigureerd waren voor het ontwikkelen met een HoloLens. Ik heb Visual Studio gepaired met de HoloLens zodat ik de applicatie kon deployen. Alsook heb ik Unity configureerd zodat deze Mixed Reality development toestaat.

Mijn eerste doel was het UI in orde brengen. Dit moest dynamisch gebeuren op basis van de gegeven data. Ik heb voor deze taak gebruikt gemaakt van een library van Microsoft genaamd de HoloToolkit.Unity. Deze library bevat voor geprogrammeerde object en UI elementen. Ik heb een Holograpic Button prefab genomen om van start te gaan, deze vormt de basis van het menu. Ik heb deze prefab aangepast zodat het design bij de use-case paste en zodat deze een url konden aanvaarden om een logo in te laden. Een menu bestaat uit meerdere van deze buttons die met behulp van het Object Collection script uitgelijnt worden. Deze menu’s worden dynamisch gegenereerd op basis van de data die de methode in kwestie ontvangt.

Het tweede doel was de connectie maken met het CRM systeem en deze dat in te laden in de applicatie. Hier heb ik erg veel problemen ondervonden omdat er leterlijk geen documentatie over het gebruik van de sdk in Unity bestaat. Uiteindelijk heb ik beslist om een met ASP.NET een wrapper API te schrijven. Deze maakt gebruikt van het XRM SDK van Microsoft om connectie te maken en alle data uit het CRM systeem te halen. In de applicatie spreekt ik deze API dan aan om de nodige data te kunnen gebruiken.

Het vierde doel was een pdf kunnen laten zien in de applicatie, hiervoor heb ik erg veel research gedaan en meerdere aanpakken geprobeert. Jammer genoeg lijkt geen enkele methode te werken. Het eindproduct is een bijna complete representatie van de bepaalde scope. Enkel het laten zien van een pdf mist. Het web API dat connecteert met het CRM systeem is een extra project geworden dat erg veel tijd heeft gevergd.
